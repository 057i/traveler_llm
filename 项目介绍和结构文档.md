# 智能旅游推荐系统 - 项目介绍

## 🎯 项目概述

### 项目名称
智能旅游推荐系统（AI-Powered Travel Recommendation System）

### 项目简介
这是一个基于**RAG（检索增强生成）**技术的智能旅游推荐系统，通过上传旅游PDF文档，自动解析、向量化并构建知识库，为用户提供个性化的旅游景点推荐和攻略问答服务。

### 核心特性
- 📄 **智能文档处理**：自动解析PDF旅游攻略，提取景点信息
- 🧠 **多路RAG检索**：稠密向量 + 稀疏向量 + 知识图谱混合检索
- 🤖 **AI智能推荐**：基于LangGraph工作流的智能推荐引擎
- 🔍 **高级搜索**：支持预算、天数、类型等多维度筛选
- 👥 **多智能体协作**：Search Agent + Analysis Agent + Synthesis Agent
- 📊 **实时进度推送**：SSE/WebSocket实时展示处理进度
- 🚀 **高性能检索**：HNSW索引 + RRF融合 + 重排序，检索延迟<100ms

### 技术亮点
- ✨ **混合检索架构**：语义相似度 + 关键词匹配，召回率提升20%
- ✨ **RRF智能融合**：多源结果融合，准确率提升15%
- ✨ **重排序优化**：BGE Reranker精排，过滤低质量结果
- ✨ **知识图谱增强**：Neo4j构建地理关系，支持"附近景点"推荐
- ✨ **工作流编排**：LangGraph可视化工作流，易扩展易维护

## 🔄 数据流向

### 1. 文档上传流程
```
用户上传PDF
    ↓
前端 (DocumentUpload组件)
    ↓
API: POST /api/documents/upload
    ↓
后端创建任务 (DocumentTaskService)
    ↓
LangGraph工作流启动 (8步处理)
    ↓
SSE实时推送进度
    ↓
前端 (WorkflowProgress组件) 展示进度
    ↓
处理完成，返回结果
```

### 2. AI推荐流程
```
用户输入查询
    ↓
前端 (AIRecommend页面)
    ↓
API: POST /api/ai-recommend/recommend
    ↓
后端启动异步工作流
    ↓
多路检索 (Milvus稠密 + Milvus稀疏 + Neo4j图谱)
    ↓
RRF融合
    ↓
重排序过滤
    ↓
LLM生成答案
    ↓
SSE推送结果
    ↓
前端展示答案和来源
```

### 3. 综合搜索流程
```
用户输入查询 + 筛选条件
    ↓
前端 (IntegratedSearch页面)
    ↓
API: POST /api/integrated-search/search
    ↓
后端构建Milvus过滤表达式
    ↓
多路检索 (带筛选)
    ↓
RRF融合
    ↓
应用层二次筛选（复杂条件）
    ↓
流式返回结果
    ↓
前端逐条展示结果卡片
```

## 🗃️ 数据库结构

### Milvus Collection Schema
```python
Collection: "travel_destinations"
├── destination_id (VARCHAR, PRIMARY KEY)    # 景点唯一ID
├── dense_vector (FLOAT_VECTOR, dim=768)     # 稠密向量
├── sparse_vector (SPARSE_FLOAT_VECTOR)      # 稀疏向量
├── name (VARCHAR)                           # 景点名称
├── province (VARCHAR)                       # 省份
├── city (VARCHAR)                           # 城市
├── category (VARCHAR)                       # 分类
├── description (VARCHAR)                    # 描述
├── rating (FLOAT)                           # 评分
├── estimated_budget (INT32)                 # 预算
├── recommended_days (INT32)                 # 推荐天数
├── travel_type (VARCHAR)                    # 旅行类型
├── tags (ARRAY<VARCHAR>)                    # 标签
└── best_season (ARRAY<VARCHAR>)             # 最佳季节

Indexes:
├── dense_vector: HNSW (M=16, efConstruction=200)
├── sparse_vector: SPARSE_INVERTED_INDEX
├── estimated_budget: STL_SORT
└── recommended_days: STL_SORT
```

### Neo4j Graph Schema
```cypher
Nodes:
├── (:Destination)    # 景点节点
│   ├── name
│   ├── province
│   ├── city
│   ├── category
│   └── description
├── (:City)           # 城市节点
│   └── name
└── (:Province)       # 省份节点
    └── name

Relationships:
├── (:Destination)-[:LOCATED_IN]->(:City)
└── (:City)-[:BELONGS_TO]->(:Province)
```

### Redis Data Structure
```
Keys:
├── doc:metadata:{doc_id}              # 文档元数据 (Hash)
├── docs:list                          # 文档ID列表 (List)
├── docs:filename:{filename}           # 文件名映射 (String)
├── document:task:{task_id}            # 任务信息 (String)
├── document:progress:{task_id}        # 任务进度 (String)
├── document:events:{task_id}          # 事件列表 (List)
├── ai:recommend:history:{session_id}  # 推荐历史 (List)
└── ai:recommend:events:{session_id}   # 推荐事件 (List)

Pub/Sub Channels:
├── document:events:{task_id}          # 文档处理事件
└── ai:recommend:events:{session_id}   # AI推荐事件
```

## 🎓 技术难点与解决方案

### 难点1: 混合检索效果不佳
**问题**: 稠密向量和稀疏向量分数尺度不同，直接融合效果差
**解决方案**: 
1. Min-Max归一化统一分数到[0,1]
2. 使用RRF基于排名融合
3. 效果提升15%

### 难点2: 重排序成本高
**问题**: CrossEncoder对所有候选重排序太慢
**解决方案**: 
1. 只对Top20重排序（而非全部）
2. 阈值过滤（score<0.5直接丢弃）
3. 延迟从2秒降到200ms

### 难点3: SSE断线重连
**问题**: 网络抖动导致进度丢失
**解决方案**: 
1. 自动重连机制（最多3次）
2. Redis持久化事件列表
3. 重连后补推未推送事件

### 难点4: 大文档解析超时
**问题**: 100页PDF解析超过10分钟
**解决方案**: 
1. 轮询间隔优化（3秒）
2. 超时时间延长（10分钟）
3. 失败自动降级（MinerU → PyMuPDF）

### 难点5: 前端大列表卡顿
**问题**: 1000条搜索结果渲染卡顿
**解决方案**: 
1. 虚拟滚动（react-window）
2. 懒加载（分页加载）
3. 渲染时间从5秒降到500ms
